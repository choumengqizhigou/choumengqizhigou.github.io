## 编程语言介绍

`C#` 作为一门编程语言，首先需要我们提炼出两个关键字，一个 **编程** ，一个 **语言** 。

**编程** 是一种行为， **语言** 是一种方式，一种手段，描述起来就像是我们编写 `C#` 代码来制作一个软件，一个功能，实现跟计算机之间的交流。

这里的语言概念跟我们日常的语言概念有很多相似的地方。

首先 **语言** 并非是单一的，生活中除了 `中文` ，还有 `英语` 、 `俄语` 、 `法语` 、 `德语` ...

计算机中呢，则有 `汇编语言` 、 `C语言` 、 `C++语言` 、 `C#语言` 、 `Java语言` 、 `Python语言` ...

其次就是语法的规则，咱们说话是要遵循一套标准、规则的，不是说随意的拼接一些词语，不然可能导致别人无法准确领会你的意思。

**遵循语法很重要，在计算机中尤为重要。**

日常用语中，你说一句 **今天中午饭吃过我了** 虽然有语法错误，别人还是能够明白你的意思。但是在计算机中，你对应语言的语法如果输错了，哪怕一个小小的符号，就会导致你无法与计算机进行交流了！

两种语言既有相似，自然也有不相似。

相似的地方这里只说一个，那就是不管你用什么编程语言，它们最后都会转换成需要执行的 `机器指令` ，机器指令最终以 `二进制` 形式存储和处理。

> 日常使用的数学是0-9，一共十个数字，简称为十进制。二进制就可以理解为0-1，只有两个数字。无论你在使用计算机时看到了什么，听到了什么，在计算机内部处理的都只是类似于010010100011这样的二进制序列。至于计算机如何将这个序列转换成图像、声音、游戏，后面咱们专门来聊。**


## 从二进制开始

从上面的描述来看，首先编程语言好像有很多种，其次不管你用什么编程语言，好像都是需要转成计算机能够处理的 `二进制` 内容。

既然这样，很直观的一个疑问立马就变成了：

**为什么需要这么多的编程语言呢？**

这个疑问其实不难解释，只不过需要咱们换个思路方向，反向考虑一下，为什么会有编程语言这个东西。

我们知道，计算机真正能够直接执行的是 **机器码** ，而机器码本质上就是一串由 `0` 和 `1` 组成的二进制数据。

不过，直接使用二进制来描述各种操作十分困难。为了帮助理解，我们先假设一种简单的编码规则：

* 0001 表示“加法”操作；
* 01 表示数字 1；
* 10 表示数字 2。

那么，如果我们想让计算机计算 `1 + 1`，就需要输入类似下面这样的二进制序列：

```
01 0001 01
```

当然，这只是一个用于说明思想的 **简化示例** ，真实计算机使用的机器码远比这复杂得多。实际的机器码不仅需要表示 **做什么操作**，还需要表示 **操作数据来自哪里** 、 **结果存放到哪里** 等信息，因此一条简单的加法指令通常都会对应更长、更复杂的二进制序列。

对于这样一个 `1 + 1` 的简单算术运算，都需要输入这么多的 `0` 和 `1` ,如果再复杂一点，岂不是要翻天了，那写代码该是多么痛苦的事情啊！

所以呢，这时候编程语言应运而生了，有一个叫做 `汇编语言` 的东西出来了。

汇编的处理逻辑很简单，要用一些人们更容易理解的概念来书写 `1 + 1`。

比如在汇编中，`add` 来表示 **加法指令** ，写起来的代码可能就是：

``` asm
mov eax,1
add eax,1
```

这样看起来是没那么困难了，不过也势必会导致另一个问题，就是你写的这些内容，计算机压根看不懂。

因为前面咱们已经交代过了，计算机只认识 `二进制` ，所以这时候呢，就需要有个转换的 **东西** ， 来将你书写的内容转换成对应的二进制序列，再交给计算机来执行。

那么这个转换的过程，就叫做 `编译` ，就是将我们写的代码转换成计算机能够识别的机器码，这个帮我们转换的 **东西** 就叫做 `编译器`。

即便引入了汇编语言，对于咱们写代码来说，也是极其低效的。

仅仅实现一些简单的功能，可能还勉强能写能看，要是涉及到一些应用开发，就需要咱们引入更多的抽象，来让越少的代码，能够做成更大的事情。 

这里先看一段汇编代码。（ **仅作演示，由chatgpt生成** ）

``` asm
section .data
    num1 db 1
    num2 db 1
    result db 0

section .text
    global _start

_start:
    mov al, [num1]
    add al, [num2]
    mov [result], al

    ; print result
    mov eax, 4
    mov ebx, 1
    mov ecx, result
    mov edx, 1
    int 0x80

    ; exit program
    mov eax, 1
    xor ebx, ebx
    int 0x80
```

上述代码使用了x86汇编语言，实现了1+1=2，并输出结果的这样一个例子。这无疑是让人头大的，属实有点太多了，绷不住啊，能不能缩减一点啊？

下面我们来请下一位选手登场，这里我们推举一下 `C语言` ，在 `C语言` 中，我们计算 `1 + 1` 可能只需要一行代码：

``` c
int a = 1 + 1;
```

无需多言，这种爽快的感觉是不是让人直接起飞！

对于咱们的 `C语言` 来说，这里不仅仅是缩减代码，语言本身还会有一些其他便利性操作或者说是设计思想，咱们可以称之为 **语言特色** 。

不同的语言也会有更适合它的应用场景，所以学习不同的语言，不仅是学习语法，更重要的是学习它们的设计思想、适用场景以及解决问题的方法。


## 语言的差异

现在我们知道代码都需要 `编译` 成本地的机器码，但你不知道是 `编译` 其实是一个比较耗时的过程，有时候哪怕只是修改一点点代码，都需要进行重新 `编译` ，所以这无疑增加了我们的等待时间，随着代码越来越多，我们等待的越来越长。既然这样有没有一种不需要 `编译` ，或者说不需要立即 `编译成本地机器码` 的方式呢？

聪明的开发人员总是能想个办法偷个懒，要不我们先写代码，然后运行的时候一行代码或者几行代码进行一次动态的 `编译` 呢？

`Python语言` 就是这其中的一个，我们称之为 **解释型语言** 。这种语言，有一个解释器，运行的时候有解释器来挨个解释代码，将代码解释为机器指令并执行，即运行到哪里，解释到哪里。（缺点就是每次运行都需要一行一行解释）

这又会导致一个新的问题，就是相比 `C/C++` 等编译型语言，`Python` 在执行效率上通常较低。一开始全部 `编译` 完成，咱们只管运行就行了，像这样运行一行,解释一行，确实会慢不少。

那有没有一种折中的方案呢，总该轮到咱们的 `C#语言` 登场了吧。

咱们的 `C#语言` 是怎么玩的呢？

它会先编译成一种中间代码 `IL` ，程序运行时，再由 `CLR` 中的 `JIT` 编译器把需要执行的部分转换成机器码。

这听起来好像也是一边运行一边解释呀？

这里有一个关键的不同就是，`C#` 第一次运行的时候，发现有些代码没有执行过，那我就编译成 `机器码` ，第二次再来运行的时候，发现这个代码已经编译过了，那我就直接调用编译之后的 `机器码` 。

有点像是 `编译+解释` 的结合体。

除此之外还有一些是为了专门便利某一行业或者某一目的做出来的编程语言，比如数值计算，图形模拟，等等等...

所以你看出来了么，编程语言就是创作者对于问题的理解，是他们想要去改善某种情况针对性开发出的一种工具。 

就跟螺丝刀有一角的 三角的 六角的。。。 有手动的，电动的， 有小马力的，大马力的。
